# <a name="state-and-data-in-docker-applications"></a>Docker 애플리케이션의 상태 및 데이터

대부분의 경우에 컨테이너를 프로세스의 인스턴스로 간주할 수 있습니다. 프로세스는 영구 상태를 유지하지 않습니다. 를 로컬 저장소 컨테이너를 작성할 수 있습니다 메모리의 단일 위치 영 속 된다고 가정 하 고 비슷합니다 인스턴스 되도록 주위 무기한으로 가정 합니다. 컨테이너 이미지를 프로세스와 같은 여러 인스턴스 가정 및는 결국 종료 합니다. 경우는 컨테이너 오 케 스트레이 터를 사용 하 여 관리 되는 경우를 가정 하는 이동 될 수 노드 또는 VM에서 다른 합니다.

다음 솔루션은 Docker 애플리케이션에서 영구 데이터를 관리하는 데 사용됩니다.

Docker 호스트에서 [Docker 볼륨](https://docs.docker.com/engine/admin/volumes/)으로:

- **볼륨** Docker에서 관리 되는 호스트 파일 시스템의 영역에 저장 됩니다.

- **탑재 바인딩** 액세스는 Docker 프로세스에서 제어할 수 없습니다 및 컨테이너를 중요 한 OS 폴더에 액세스할 수 없습니다 보안상 위험할 수 있으므로 호스트 파일 시스템의 폴더를 매핑할 수 있습니다.

- **tmpfs 탑재**는 호스트 메모리에만 존재하고 파일 시스템에 기록되지 않는 가상 폴더와 같습니다.

원격 스토리지에서:

- [Azure Storage](https://azure.microsoft.com/documentation/services/storage/) 컨테이너에 대 한 적합 한 장기 지 속성 솔루션을 제공 하는 지역 배포 가능한 저장소를 제공 합니다.

- 같은 원격 관계형 데이터베이스 [Azure SQL Database](https://azure.microsoft.com/services/sql-database/), 같은 NoSQL 데이터베이스 [Azure Cosmos DB](https://docs.microsoft.com/azure/cosmos-db/introduction), 또는 같은 서비스 캐시 [Redis](https://redis.io/)합니다.

Docker 컨테이너에서:

- Docker는 *오버레이 파일 시스템*이라는 기능을 제공합니다. 이 기능은 저장소 컨테이너의 루트 파일 시스템에 정보를 업데이트 하는 쓰기 시 복사 작업을 구현 합니다. 해당 정보 "레이아웃의 상단에서" 컨테이너의 기반이 되는 원본 이미지입니다. 시스템에서 컨테이너를 삭제하는 경우 해당 변경 내용이 손실됩니다. 따라서 로컬 저장소 내 컨테이너의 상태를 저장할 수 있지만, 기본적으로 상태 비저장 인 컨테이너 디자인의 전제와 충돌이 기능을 기반으로 시스템을 디자인 합니다.

- 그러나 Docker 볼륨 Docker에서 로컬 데이터를 처리 하는 기본 방법은 되었습니다. 저장소 컨테이너에 대 한 자세한 내용은, 필요한 경우 확인할 [Docker 저장소 드라이버](https://docs.docker.com/engine/userguide/storagedriver/) 하 고 [이미지, 컨테이너 및 저장소 드라이버에 대 한](https://docs.docker.com/engine/userguide/storagedriver/imagesandcontainers/)합니다.

다음 이러한 옵션에 대 한 추가 정보를 제공 합니다.

**볼륨**은 컨테이너에 있는 호스트 OS에서 디렉터리에 매핑된 디렉터리입니다. 컨테이너의 코드에 디렉터리에 대한 액세스 권한이 있는 경우 실제로 호스트 OS의 디렉터리에 액세스할 수 있습니다. 이 디렉터리는 컨테이너 자체의 수명과 관련이 없으며, 디렉터리는 Docker에 의해 관리되고 호스트 머신의 핵심 기능과 격리되어 있습니다. 따라서 데이터 볼륨은 컨테이너의 수명과 독립적으로 데이터를 유지하도록 설계되었습니다. Docker 호스트에서 컨테이너 또는 이미지를 삭제하는 경우 데이터 볼륨에서 유지되는 데이터는 삭제되지 않습니다.

볼륨은 이름을 지정하거나 익명화할 수 있습니다(기본값). 명명된 볼륨은 **데이터 볼륨 컨테이너**의 발전이며 컨테이너 간에 데이터를 쉽게 공유할 수 있도록 해줍니다. 볼륨도 다른 옵션과 함께 원격 호스트에서 데이터를 저장할 수 있도록 하는 볼륨 드라이버를 지원 합니다.

**탑재 바인딩** 오랜 시간 동안 제공 되었습니다 및 컨테이너에 탑재 지점 폴더의 매핑을 허용 합니다. 탑재 바인딩은 볼륨보다 더 많은 제한 사항과 몇 가지 중요한 보안 문제가 있으므로 볼륨이 권장되는 옵션입니다.

**`tmpfs` 탑재** 는 가상 폴더는 호스트의 메모리에만 라이브 및 파일 시스템에 기록 되지 않습니다. 이들은 빠르고 안전하지만 메모리를 사용하며 비영구적 데이터에만 사용됩니다.

그림 4-5와 같이 일반 Docker 볼륨은 호스트 서버 또는 VM의 물리적 장벽 내부가 아닌 컨테이너 외부에 저장될 수 있습니다. 그러나 Docker 컨테이너는 호스트 서버 또는 VM 간의 볼륨에 액세스할 수 없습니다. 즉, 이러한 볼륨 이므로 없이 원격 호스트를 지 원하는 볼륨 드라이버를 사용 하 여 설정할 수 있지만 여러 Docker 호스트에서 실행 되는 컨테이너 간에 공유 된 데이터를 관리할 수 있습니다.

![볼륨을 컨테이너 간에 공유할 수 있지만 원격 호스트를 지원하는 원격 드라이버를 사용하지 않는 한 동일한 호스트에서만 공유할 수 있습니다. ](./media/image5.png)

**그림 4-5** 컨테이너 기반 애플리케이션에 대한 볼륨 및 외부 데이터 원본

또한 오케스트레이터에서 Docker 컨테이너를 관리할 때 컨테이너는 클러스터에서 수행한 최적화에 따라 호스트 간에 "이동"할 수 있습니다. 따라서 비즈니스 데이터에 데이터 볼륨을 사용하는 것을 권장하지 않습니다. 하지만 추적 파일, 임시 파일을 사용 하 여 작업 하는 적절 한 메커니즘이 되거나 마찬가지로 영향을 주지 않는 비즈니스 데이터 일관성.

컨테이너 없이 개발할 때 사용하는 것과 동일한 방식으로 Azure SQL Database와 같은 **원격 데이터 원본 및 캐시** 도구, Azure Cosmos DB 또는 Redis와 같은 원격 캐시를 컨테이너화된 애플리케이션에서 사용할 수 있습니다. 비즈니스 애플리케이션 데이터를 저장하는 검증된 방법입니다.

**Azure Storage.** 비즈니스 데이터는 일반적으로 외부 리소스 또는 Azure Storage와 같은 데이터베이스에 배치 해야 합니다. Azure Storage는 클라우드의 다음 서비스를 제공 합니다.

- Blob 스토리지는 구조화되지 않은 개체 데이터를 저장합니다. Blob는 문서 또는 미디어 파일(이미지, 오디오 및 비디오 파일)과 같은 텍스트 또는 이진 데이터 형식일 수 있습니다. Blob 스토리지를 개체 스토리지라고도 합니다.

- File storage는 표준 SMB 프로토콜을 사용 하 여 레거시 응용 프로그램을 위한 공유 저장소를 제공 합니다. Azure 가상 머신 및 클라우드 서비스는 탑재된 공유를 통해 애플리케이션 구성 요소 간에 파일 데이터를 공유할 수 있습니다. 온-프레미스 응용 프로그램에는 파일 공유의 데이터에는 파일 서비스 REST API를 통해 액세스할 수 있습니다.

- 테이블 스토리지는 구조화된 데이터 세트를 저장합니다. 테이블 스토리지는 NoSQL 키 특성 데이터 저장소입니다. 이를 통해 많은 양의 데이터를 신속하게 개발하고 빠르게 액세스할 수 있습니다.

**관계형 데이터베이스 및 NoSQL 데이터베이스** SQL Server, PostgreSQL, Oracle과 같은 관계형 데이터베이스 또는 Azure Cosmos DB, MongoDB와 같은 NoSQL 데이터베이스 등 다양한 외부 데이터베이스가 있습니다. 이러한 데이터베이스는 완전히 다른 항목을 사용 되기 때문에이 가이드의 일부로 설명 하지 않습니다.
