# <a name="base-classes-for-implementing-abstractions"></a><span data-ttu-id="f40f4-102">추상화 구현을 위한 기본 클래스</span><span class="sxs-lookup"><span data-stu-id="f40f4-102">Base Classes for Implementing Abstractions</span></span>
<span data-ttu-id="f40f4-103">엄격히 말해, 클래스는 다른 클래스에서 파생 됩니다 하는 경우 기본 클래스를 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="f40f4-103">Strictly speaking, a class becomes a base class when another class is derived from it.</span></span> <span data-ttu-id="f40f4-104">그러나이 섹션 기본 클래스는 클래스 주로 일반적인 추상화를 제공 하거나 일부를 재사용 하는 다른 클래스에 대 한 기본 구현 하지만 상속 하도록 설계 되었습니다.</span><span class="sxs-lookup"><span data-stu-id="f40f4-104">For the purpose of this section, however, a base class is a class designed mainly to provide a common abstraction or for other classes to reuse some default implementation though inheritance.</span></span> <span data-ttu-id="f40f4-105">기본 클래스는 일반적으로 중간 계층의 루트에 추상화 사이의 맨 아래에 몇 가지 사용자 지정 구현이 상속 계층 구조 배치 합니다.</span><span class="sxs-lookup"><span data-stu-id="f40f4-105">Base classes usually sit in the middle of inheritance hierarchies, between an abstraction at the root of a hierarchy and several custom implementations at the bottom.</span></span>  
  
 <span data-ttu-id="f40f4-106">추상화 구현을 위한 도우미 구현으로 사용 합니다.</span><span class="sxs-lookup"><span data-stu-id="f40f4-106">They serve as implementation helpers for implementing abstractions.</span></span> <span data-ttu-id="f40f4-107">예를 들어, 순서가 지정 된 컬렉션 항목에 대 한 프레임 워크의 추상화 중 하나는 <xref:System.Collections.Generic.IList%601> 인터페이스입니다.</span><span class="sxs-lookup"><span data-stu-id="f40f4-107">For example, one of the Framework’s abstractions for ordered collections of items is the <xref:System.Collections.Generic.IList%601> interface.</span></span> <span data-ttu-id="f40f4-108">구현 <xref:System.Collections.Generic.IList%601> 하며, 따라서 프레임 워크를 제공 몇 가지 기본 클래스와 같은 <xref:System.Collections.ObjectModel.Collection%601> 및 <xref:System.Collections.ObjectModel.KeyedCollection%602>, 사용자 지정 컬렉션을 구현 하기 위한 도우미 역할도 하는 합니다.</span><span class="sxs-lookup"><span data-stu-id="f40f4-108">Implementing <xref:System.Collections.Generic.IList%601> is not trivial, and therefore the Framework provides several base classes, such as <xref:System.Collections.ObjectModel.Collection%601> and <xref:System.Collections.ObjectModel.KeyedCollection%602>, which serve as helpers for implementing custom collections.</span></span>  
  
 <span data-ttu-id="f40f4-109">너무 많은 구현을 포함 하는 경향이 있으므로 기본 클래스 추상화로 단독으로 제공 하는 데 적합 하지 일반적으로 합니다.</span><span class="sxs-lookup"><span data-stu-id="f40f4-109">Base classes are usually not suited to serve as abstractions by themselves, because they tend to contain too much implementation.</span></span> <span data-ttu-id="f40f4-110">예를 들어 합니다 `Collection<T>` 기본 클래스 구현 제네릭이 아닌 구현 하는 팩트에 관련 된 많은 포함 `IList` 인터페이스 (제네릭이 아닌 컬렉션을 사용 하 여 더 효율적으로 통합) 및 된다는 것에 저장 된 항목의 컬렉션 해당 필드 중 하나에 메모리입니다.</span><span class="sxs-lookup"><span data-stu-id="f40f4-110">For example, the `Collection<T>` base class contains lots of implementation related to the fact that it implements the nongeneric `IList` interface (to integrate better with nongeneric collections) and to the fact that it is a collection of items stored in memory in one of its fields.</span></span>  
  
 <span data-ttu-id="f40f4-111">앞에서 설명한 대로 기본 클래스 추상화를 구현 해야 하는 사용자에 대 한 매우 유용한 도움말을 제공할 수 있지만 동시에 상당한 부담으로 작용할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="f40f4-111">As previously discussed, base classes can provide invaluable help for users who need to implement abstractions, but at the same time they can be a significant liability.</span></span> <span data-ttu-id="f40f4-112">노출 영역을 추가 및 상속 계층의 깊이 증가 하며 나타내므로 개념적 프레임 워크를 복잡 하 게 합니다.</span><span class="sxs-lookup"><span data-stu-id="f40f4-112">They add surface area and increase the depth of inheritance hierarchies and so conceptually complicate the framework.</span></span> <span data-ttu-id="f40f4-113">따라서 프레임 워크의 사용자에 게 중요 한 가치를 제공 하는 경우에 기본 클래스를 사용 해야 합니다.</span><span class="sxs-lookup"><span data-stu-id="f40f4-113">Therefore, base classes should be used only if they provide significant value to the users of the framework.</span></span> <span data-ttu-id="f40f4-114">프레임 워크에는 기본 클래스에서 상속 하는 대신 내부 구현 사례 위임 강력 하 게 고려해 야의 구현자에만 값을 제공 하는 경우 피해 야 합니다.</span><span class="sxs-lookup"><span data-stu-id="f40f4-114">They should be avoided if they provide value only to the implementers of the framework, in which case delegation to an internal implementation instead of inheritance from a base class should be strongly considered.</span></span>  
  
 <span data-ttu-id="f40f4-115">**✓ CONSIDER** 함으로써 기본 추상 멤버가 포함 하지 않는 경우에 추상 클래스입니다.</span><span class="sxs-lookup"><span data-stu-id="f40f4-115">**✓ CONSIDER** making base classes abstract even if they don’t contain any abstract members.</span></span> <span data-ttu-id="f40f4-116">이 사용자에 게 명확 하 게 통신 클래스에서 상속 하는 용도로 설계 됩니다.</span><span class="sxs-lookup"><span data-stu-id="f40f4-116">This clearly communicates to the users that the class is designed solely to be inherited from.</span></span>  
  
 <span data-ttu-id="f40f4-117">**✓ CONSIDER** 주요 시나리오 형식에서 별도 네임 스페이스에 기본 클래스를 배치 합니다.</span><span class="sxs-lookup"><span data-stu-id="f40f4-117">**✓ CONSIDER** placing base classes in a separate namespace from the mainline scenario types.</span></span> <span data-ttu-id="f40f4-118">정의상, 기본 클래스는 고급 확장성 시나리오를 위한 것 및 따라서는 대부분의 사용자와 관련이 없습니다.</span><span class="sxs-lookup"><span data-stu-id="f40f4-118">By definition, base classes are intended for advanced extensibility scenarios and therefore are not interesting to the majority of users.</span></span>  
  
 <span data-ttu-id="f40f4-119">**X AVOID** 클래스가 공용 Api에서 사용 하기 위한 경우 "기본" 접미사를 사용 하 여 기본 클래스 이름을 지정 합니다.</span><span class="sxs-lookup"><span data-stu-id="f40f4-119">**X AVOID** naming base classes with a "Base" suffix if the class is intended for use in public APIs.</span></span>
